### Scoring (services/scoring.js)

**Technical Health (0-100):** Weighted sum of 8 checks (excludes speed). Weights: ssl=15, viewport=10, meta=17, h1=10, schema=13, social=10, contact=11, content=14. Formula: check.score * (weight/10), summed, capped at 100.

**AI Visibility (0-100):** 7 signals: faqSchema(15), localBusiness(15), blog(10), gbp(15), reviews(15), placesListing(10), aiMentions(20).

### Issues flagged

1. **META CHECK — "perfect" page scores 5/10, not 10/10.** The fixture-1 perfect page has title=76 chars (outside 20-70 range) and description=186 chars (outside 50-160 range). The scoring thresholds are VERY tight — a real well-optimised page fails because the ranges don't match real-world best practice. Title 70ch cap is too low (Google shows ~60 chars but allows up to 70 in SERPs); description 160ch cap is too low (Google allows ~155-160, but many good descriptions are 160-200). **BLOCKER: sells wrong "fix" to clients.**

2. **CONTENT CHECK — viewport gives 10 points for bare minimum.** The Ascari-style page with just `<meta name="viewport">` and almost no content scores 31/100 (ssl=10, viewport=10, speed=10, content=1). A page with zero useful signals still gets 31 points. This inflates the score and reduces the urgency of the fix recommendation. **HIGH: reduces conversion pressure.**

3. **NO DUPLICATE META DETECTION.** The detector doesn't check for or penalise duplicate `<meta name="description">` tags. Two conflicting descriptions confuse search engines and AI. This is a known SEO issue the Detective should flag. **MEDIUM: missing detection.**

4. **SPEED CHECK BASED ON HTML SIZE ONLY.** The `speed` check measures `Buffer.byteLength(html)` — this is the raw HTML document size, not the rendered page weight. A page loading 5MB of JavaScript but with small HTML scores 10/10. The speed check is excluded from Technical Health scoring (deliberately), but its presence in the overallScore (the AEO detector's own 0-100) is misleading. **LOW: excluded from dual scoring already.**

## Reporter Agent (jobs/scheduledReports.js + services/weeklyProDigest.js)

### What the Reporter produces

The Monday 08:00 UTC cron (`'0 8 * * 1'`) calls `buildWeeklyProDigestForAllVendors()`, then for each Pro vendor:
1. Writes a `WeeklyReport` snapshot via `findOrCreate` (idempotent)
2. Sends the digest email via `weeklyProDigestTemplate`
3. Writes an `AgentRun` record with `agentName: 'reporter'`

### Digest sections built

| Section | Data source | Query |
|---------|-------------|-------|
| score | AeoReport (last 50, grouped by week) | current/previous/delta/12-week history |
| citations | AIMentionScan (this week) | total, byPlatform (6 platforms), captured array |
| agentActivity | AgentRun (this week, all 5 agents) | ran/drafts/findings/submitted/sent/queriesScanned |
| needsAttention | ApprovalQueue (pending for vendor) | itemType, title |
| competitorMoves | AIMentionScan (this week, mentioned=false) | competitors cited on queries vendor missed |

### Issues flagged

5. **citations.byPlatform.change is always 0.** The digest builds `change: 0` for every platform (line 82 of weeklyProDigest.js). It never computes the delta vs last week. The frontend renders "no change" for every platform every week. **BLOCKER: the per-engine comparison chart (PlatformBreakdownSection.tsx) is always flat.**

6. **score.current may be null indefinitely for new Pro vendors.** If a vendor has never had an AeoReport generated, `score.current` stays null forever. The digest handles this with "being calculated" text, but no mechanism triggers the initial AeoReport generation for a newly upgraded Pro vendor. The old weekly cron called `generateFullReport()` which would have created one — the new digest cron does not. **BLOCKER: new Pro customers see "being calculated" forever.**

7. **Starter cron still calls deprecated generateVendorReports().** The Starter monthly cron at `'0 6 1 * *'` calls the old function which inlines an email with `vendor.company` (not `vendor.name`). This works but the inline email template is a minimal 3-line HTML without TendorAI branding. **LOW: Starter is unused currently (0 Starter customers).**

### Issues flagged

8. **FABRICATION DETECTION GAP — unsourced stats in body not flagged as errors.** The validator's `findUnsourcedStats` runs and produces warnings, but warnings don't block publish. A post claiming "87% of solicitors fail to comply" with no source citation passes validation and goes to the approval queue marked as passed. Admin must manually catch fabricated stats. **HIGH: reputation risk if admin misses it.**

9. **Monthly per-vendor cap is 4 but cron now runs 3x/week.** With the Writer firing Mon/Wed/Fri (PR #49), a vendor could receive 3 drafts in week 1 and hit the cap after 1 draft in week 2, producing an uneven distribution. The cap counts calendar-month completed runs, not per-week. In a 4-week month with 3x/week schedule, that's 12 potential runs but only 4 allowed — the vendor gets drafts Mon/Wed of week 1, Mon of week 2, then nothing for the rest of the month. **MEDIUM: customer perception of inconsistent value.**

10. **Cost cap check runs AFTER the Claude API call.** Line 219-228 of writerAgent.js: the cost cap is checked after `response` is returned from Claude. If the platform cost is already at $74.90 and this call costs $0.15, the money is already spent — the cap check just fails the run and wastes the spend. **MEDIUM: money lost on cap-boundary runs.**
